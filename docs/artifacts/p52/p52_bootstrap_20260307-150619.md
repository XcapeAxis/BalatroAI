## Current Known Gaps For A Learned Router

1. No supervised router dataset builder exists yet; P48 traces and P39/P41 summaries are still disconnected.
2. Existing P48 routing traces are rule-decision traces, not controller-outcome traces; label generation must join state traces with arena/ablation priors and mark confidence.
3. Arena outputs are mostly aggregate per controller / per bucket, so P52 v1 will need mixed-strength labels instead of strict per-state optimal labels.
4. Current hybrid trace schema does not expose router mode, guard trigger, learned probabilities, or OOD fallback reasons.
5. P41 triage schema has hybrid routing fields, but they are not yet populated for learned-router comparisons.
6. P51 registry and campaign stages do not recognize `family=learned_router`.
7. P22 config/orchestrator has no router dataset build, learned router train, or router ablation experiment types.
8. Dashboard has no learned-router panels yet.

## Immediate Implementation Direction

- Reuse existing routing feature schema as the v1 learned-router input schema.
- Build router labels by combining:
  - direct rule-router choice from P48 traces
  - controller rankings from P48/P39 arena summaries
  - campaign/checkpoint references from P41/P51
- Treat labels as confidence-weighted and provenance-aware:
  - `arena`
  - `rule`
  - `heuristic`
  - `mixed`
- Keep rule router as the safety floor and add learned fallback only behind explicit router modes and rule guard logging.
